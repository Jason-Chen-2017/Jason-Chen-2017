第 1 章 系统概述与架构设计

1.1 系统简介

本系统是基于 OpenClaw 多 Agent 框架 与 Claude Code AI 编程助手 构建的端到端研发自动化平台，实现从需求分析到生产部署的全流程自动化。系统支持各研发角色的岗位 Agents 协同工作，并在关键节点提供人机协同能力。

✨ 核心优势：
  • 🚀 研发效率提升 10 倍以上
  • 🎯 全流程自动化，减少人工干预
  • 🔒 质量门禁保障代码质量
  • 👥 人机协同，灵活可控
  • 📊 全流程可视化追踪

2.2.2 Agent 添加命令

# 添加子 Agent
openclaw agents add product-agent --workspace ~/.openclaw/workspace-product

# 配置主 Agent 调度权限
openclaw config set agents.list[0].subagents.allowAgents '["product-agent","backend-architect","frontend-architect"]' --json

# 为子 Agent 指定轻量模型（节省成本）
openclaw config set agents.list[5].model 'deepv-easyclaw/claude-haiku-4-5' --json

# 验证配置
openclaw agents list
openclaw channels status
⚠️ 重要注意事项：
  • 每个 Agent 必须有独立 workspace，绝不能共用
  • 必须为子 Agent 编写专属 AGENTS.md，不能复制主 Agent 配置
  • 多 Agent 配置支持热重载，无需重启 Gateway
  • 配置前自动切换到 Sonnet 模型，避免轻量模型配置风险

3.4 人机协同评审流程

👥 人机协同节点：PRD 评审

Agent 完成工作：

  • Product-Agent 生成 PRD 初稿
  • 自动检查 PRD 完整性（必填字段、验收标准等）
  • 生成评审问题清单

人工介入工作：

  • 产品经理审核 PRD 内容准确性
  • 技术负责人评估技术可行性
  • 业务方确认需求优先级
  • 在飞书/钉钉中直接评论批注

反馈循环：

  1. 人工评审意见通过 IM 工具反馈给 Orchestrator-Agent
  2. Orchestrator-Agent 将修改意见转发给 Product-Agent
  3. Product-Agent 根据反馈修订 PRD
  4. 重复评审流程直到通过

CREATE TABLE users (

    status TINYINT DEFAULT 0,

import lombok.RequiredArgsConstructor;
import org.springframework.http.HttpStatus;
import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.annotation.*;

import jakarta.validation.Valid;

@RestController
@RequestMapping("/api/v1/users")
@RequiredArgsConstructor
public class UserController {
    
    private final UserService userService;
    
    /**
     * 用户注册接口
     * @param request 注册请求体
     * @return 用户信息
     */
    @PostMapping("/register")
    public ResponseEntity register(
            @Valid @RequestBody UserRegisterRequest request) {
        UserResponse response = userService.register(request);
        return ResponseEntity.status(HttpStatus.CREATED).body(response);
    }
    
    @GetMapping("/{id}")
    public ResponseEntity getUser(@PathVariable Long id) {
        UserResponse response = userService.getById(id);
        return ResponseEntity.ok(response);
    }
}

                }
            }

                }
            }
        }

                        docker.image("${REGISTRY}/${IMAGE_NAME}:${BUILD_ID}").push('latest')

                }

        }
        failure {
            echo '部署失败，执行回滚！'
            sh '''
                kubectl rollout undo deployment/user-service -n ecommerce
            '''
        }
    }
}

11.2 最佳实践总结

11.2.1 Agent 配置最佳实践

  • 模型选择: 复杂任务用 Sonnet/Opus，简单任务用 Haiku 节省成本
  • Workspace 隔离: 每个 Agent 独立 workspace，避免上下文污染
  • 人设定义: 明确的 AGENTS.md 和 SOUL.md，防止角色混淆
  • 权限控制: 通过 subagents.allowAgents 精确控制调度权限

11.2.2 人机协同最佳实践

  • 关键节点审批: PRD、技术方案、UAT 验收必须人工审批
  • 超时处理: 设置合理的审批超时时间，避免流程阻塞
  • 反馈闭环: 人工评审意见通过 IM 工具快速反馈给 Agent
  • 渐进式自动化: 从简单项目开始，逐步扩大自动化范围

11.2.3 质量保障最佳实践

  • 多层质量门禁: 规划质量→开发质量→生产就绪三重检查
  • 测试左移: 在代码生成阶段就编写单元测试
  • 持续监控: 部署后持续监控应用性能和业务指标
  • 快速回滚: 部署失败自动回滚到上一个稳定版本

11.3 常见问题 FAQ

Q1: Agent 自称"我是主 Agent"怎么办？

A: 这是复制了主 Agent 的 AGENTS.md 导致的。必须为子 Agent 编写专属 AGENTS.md，明确说明"我是主 Agent 的 XX 助手，不是主 Agent"。

Q2: 多 Agent 配置后 Gateway 无响应？

A: 检查是否误删了 channels.feishu 中的 allowFrom、dmPolicy、groupPolicy 字段。这些字段必须保留，否则健康监控会 stuck。

Q3: 如何降低多 Agent 运行成本？

A: 为简单任务的子 Agent 配置轻量模型 (Haiku/Flash/mini)，可以节省 3-5 倍成本。例如 API 设计、单元测试等任务可以用 Haiku。

Q4: 人机协同审批超时如何处理？

A: 在审批配置中设置 timeout_hours 和 escalation 策略。可以选择超时自动跳过、升级到上级审批人或发送紧急通知。

Q5: 代码生成质量不高怎么办？

A: 提高质量门禁阈值，启用自动修复机制。对于复杂业务逻辑，可以让人工先编写伪代码或详细注释，再让 Agent 实现。
